## 🚀 프로젝트 개요 (Overview)

**Withy**는 '함께 보는 즐거움'을 목표로 기획된 지연 없는 OTT 영상 같이보기(Watch-Party) 서비스입니다. 기존 화상회의 솔루션이나 화면 공유 기술에서 발생하는 시청자와 방장 간의 **영상 재생 타임라인 불일치** 문제를 해결하기 위해 고안되었습니다.

뿐만 아니라 실시간 채팅 시 발생할 수 있는 스포일러와 악성 채팅을 즉시 차단하는 **클린 채팅 환경**을 결합하여 독보적인 시청자 경험을 제공합니다.

---

## 🏗️ 핵심 시스템 아키텍처 (Architecture)

### 1. 실시간 영상 동기화 (Timeline Sync)
* **WebSocket**을 통한 양방향 소켓 통신으로 방장과 참여자 간의 재생(Play), 일시정지(Pause), 탐색(Seek) 이벤트를 **밀리세컨드 단위로 조율**합니다.
* 영상 버퍼링으로 인한 오차를 줄이기 위해 클라이언트 간 상태 오차를 계산 클라이언트 측에서 부드럽게 보정합니다.

### 2. 욕설 및 스포일러 차단 통신망 (AI Clean Chat)
* 사용자가 채팅을 입력하면 WebSocket 서버가 **Kafka** 메시지 큐를 통해 AI 분석 서버(Python)로 즉각적인 채팅 필터링을 요청합니다.
* AI 모듈과 백엔드 서버는 **gRPC**를 통해 통신하여 직렬화 병목을 최소화하고, 수 밀리초 내로 해당 메시지의 스포일러/욕설 여부를 회신합니다.
* 정상 메시지일 경우만 채팅방에 브로드캐스팅되어 클린한 환경이 유지됩니다.

## 💻 사용 기술 스택 (Tech Stack)

### Frontend
- **React**: 컴포넌트 기반 UI 설계 및 전역 상태 관리
- **WebRTC / WebSocket**: 실시간 영상 동기화 및 채팅 구현
- **TailwindCSS**: 반응형 및 신속한 UI 스타일링

### Backend
- **Spring Boot**: 견고한 REST API 및 소켓 메인 서버
- **gRPC**: 마이크로서비스 간 고속 통신 (채팅 필터링 파이프라인)
- **Kafka**: 대규모 채팅 트래픽 분산 처리 및 메시지 큐
- **Redis**: 실시간 유저 접속 상태 및 세션 정보 인메모리 관리
- **Python (AI)**: 자연어 처리 기반 채팅 필터링 모델

## 🧐 기술적 해결 과정 (Challenges & Solutions)

> **문제점**: 채팅이 입력될 때마다 AI 분석 서버를 거치면, 트래픽이 몰릴 시 엄청난 지연(Lag)이 발생하여 '실시간 채팅'의 본질이 훼손됩니다.

**해결 방안**: HTTP REST API 대신 **gRPC**와 **Kafka**를 도입했습니다. 
스트리밍 방식으로 데이터를 직렬화하여 패킷 크기를 혁신적으로 줄였으며, Kafka를 통해 메시지 폭주 시 큐잉 처리로 서버 다운을 방지했습니다. 결과적으로 AI 분석을 포함하고도 메시지 전송 지연시간을 50ms 이내로 단축하는 데 성공하였습니다.
